## Part 2 — What already works in `mobile/` (do not regress)

These were just fixed before the move. Test the flow end-to-end before adding new features.

1. **Pairing.** `app/(onboarding)/pair.tsx` → `POST /api/pair/exchange` → SecureStore.
2. **Health Connect permissions on Android 14+.** Three native pieces, all wired by `plugins/with-health-connect-delegate.js`:
   - `MainActivity.onCreate` registers `HealthConnectPermissionDelegate.setPermissionDelegate(this)` (fixes a `lateinit` crash on tap).
   - `AndroidManifest.xml` has `<queries>` for `com.google.android.apps.healthdata` and `com.google.android.healthdata` (without these, Android 11+ silently no-ops the permission intent).
   - `MainActivity` has the Android 14 `VIEW_PERMISSION_USAGE` + `HEALTH_PERMISSIONS` intent filter (without it, `PermissionsActivity` logs `App should support rationale intent, finishing!` and closes).
3. **Bottom-sheet handoff UX.** `app/(onboarding)/permissions.tsx` shows step-by-step instructions, opens Health Connect via `requestPermissions()`, and uses an `AppState` listener to detect return-from-settings and verify the grant.
4. **Auth guard never goes stale.** `app/_layout.tsx` re-reads SecureStore on segment changes and on app foreground, so a fresh pairing or a permission grant doesn't bounce the user back to `/(onboarding)/pair`.
5. **Vitals sync error envelope.** `src/sync/client.ts` accepts both `{"error": "..."}` and the FastAPI-wrapped `{"detail": {"error": "..."}}`. **Make sure** `sentinel-live/docs/backend-contract.md` documents this — if it doesn't, port the comment block from this file's `readErrorCode` helper into the doc.
6. **`sentinel://` deep link with no path is caught.** `app/index.tsx` redirects to `/(main)/status` so Health Connect's redirect-back doesn't show "unmatched route."

**Acceptance check** after the merge: pair a fresh device, grant Health Connect permission, walk for a couple of minutes, confirm `GET /api/patients/{pid}/vitals` returns samples on the backend.

## Part 3 — Build the patient dashboard + call receiving

### 3.1 The patient dashboard, ported from `frontend/app/patient/`

The web view at `sentinel-live/frontend/app/patient/page.tsx` + `components/patient/PatientLiveView.tsx` is the spec for the mobile screen. Read those two files before starting. Concretely it does:

- Fetches patient name + recent calls server-side (`api.patients()`, `api.calls(pid)`).
- Subscribes to `GET /api/stream` (SSE) via `useEventStream`.
- Renders a "● live" / "● connecting" badge driven by SSE connection state.
- On `pending_call` for this patient, shows a green "Sentinel is calling you" panel with **Answer** / **Dismiss**.
- On Answer, mounts the `<elevenlabs-convai agent-id="...">` web component for the conversation.
- On `call_scored`, refetches calls and re-renders the latest `summary` + a trajectory chart of `score.deterioration` over time.
- Highlights the latest panel red when `recommended_action === "suggest_911"`.

Mobile equivalents to build under `app/(main)/`:

| Web file | Mobile target | Notes |
|---|---|---|
| `frontend/app/patient/page.tsx` | `app/(main)/status.tsx` (extend the existing one) | Server-side `getSession` becomes "read SecureStore + call `GET /api/patients/{pid}` directly." |
| `components/patient/PatientLiveView.tsx` | `app/(main)/status.tsx` UI | Same layout, RN primitives instead of Tailwind divs. |
| `lib/hooks/useEventStream.ts` | `src/realtime/useEventStream.ts` (NEW) | Use `react-native-sse` (works in Hermes; the browser `EventSource` does not exist in RN). Reuse the same connection-with-backoff logic. Auth: SSE in this backend is cookie-gated (`withCredentials: true`); the mobile app uses the device JWT. **Confirm with the backend agent** whether `/api/stream` will accept `Authorization: Bearer <device_token>` for mobile. If not, add a `?token=` query param accepted on the SSE endpoint, or open a dedicated `/api/stream/device` route. This is a backend change — do not silently work around it. |
| `lib/api.ts` | `src/api/client.ts` (extend the existing sync client) | Add `getPatient(pid)`, `getCalls(pid)`, `getAlerts()`. Reuse the device JWT and the `readErrorCode` helper from `src/sync/client.ts`. |
| `components/TrajectoryChart` | `react-native-svg-charts` or `victory-native` | The web one uses Recharts; pick one RN charting lib and stick with it. |

Out of scope for the mobile patient view (these stay clinician-only on the web `/admin`): the patient grid, KPI strip, alert feed, cohort similarity panel, "Call now" button.

### 3.2 Receiving calls — the right answer is the official ElevenLabs React Native SDK

The web view uses ElevenLabs' Convai widget, which is a browser web component. It does **not** work in React Native. There are two valid approaches; the first is strongly recommended:

#### Option A (recommended): `@elevenlabs/react-native`

ElevenLabs ships an official React Native SDK that re-exports the React conversation API and pre-wires WebRTC + native AudioSession via LiveKit. As of April 2026 it's at v1.1.1 and actively maintained. Install:

```bash
npm install @elevenlabs/react-native @livekit/react-native @livekit/react-native-webrtc livekit-client
```

Constraints:
- **Requires Expo development builds.** Expo Go won't work because of the LiveKit native modules. This is fine — this app is already on a dev build for Health Connect.
- **Microphone permission required.** Add `NSMicrophoneUsageDescription` to `app.json` `ios.infoPlist` and `RECORD_AUDIO` to `android.permissions`. The plugin already sets a few permissions; extend it.
- **Cannot run on iOS Simulator without audio routing fixes** — test on a real device.

Wiring: when the SSE handler receives `pending_call`, set state to show the "Sentinel is calling you" panel. On Answer, call `useConversationControls().startSession({ agentId })`. Render conversation status from `useConversationStatus`. End on user tap or remote disconnect.

The `agent-id` lives in `frontend/.env.local` as `NEXT_PUBLIC_ELEVENLABS_AGENT_ID`. Mirror it as `EXPO_PUBLIC_ELEVENLABS_AGENT_ID` in `mobile/.env` and reference via `process.env.EXPO_PUBLIC_ELEVENLABS_AGENT_ID`.

#### Option B (fallback): Convai widget in a `react-native-webview`

If the official SDK has integration issues, drop a tiny HTML page hosted by the backend (or inlined as a string) that loads the Convai widget script, and render it in a WebView. Worse audio quality and worse lifecycle handling, but takes 30 minutes. Use only if Option A blocks the demo.

#### What the mobile app does **not** need (for the hackathon)

- **No CallKit / PushKit / `react-native-callkeep` / FCM VoIP push.** Those are needed only if you want the system call UI to ring while the app is killed. The hackathon flow is "patient has the app open or in the foreground; clinician taps Call Now; SSE delivers `pending_call`; in-app green banner appears." The web `/patient` view works the same way and that's what's being demoed today.
- **No background SSE.** SSE in this app pauses when backgrounded; reconnect-on-foreground (which `useEventStream` already does via the `AppState` listener you're going to port) is enough.

If anyone asks for "ring the phone like a real call when the app is killed," that's a v2 conversation — it requires a Twilio number, FCM data messages, an Android foreground service, iOS PushKit credentials, App Store call-screening review, and `react-native-callkeep`. Don't start it without scoping a separate sprint.

### 3.3 Acceptance criteria for the mobile patient dashboard

The mobile patient screen ships when:

1. After pairing + permissions, opening the app lands on a screen showing the patient's name, a "● live" badge, and a list of recent check-ins.
2. With the backend running and an admin clicking **Call Now** on the web `/admin` for this patient, within 1 second the mobile screen shows a green "Sentinel is calling you" panel.
3. Tapping **Answer** starts an ElevenLabs conversation (audio in/out works on a real device).
4. After the call ends, within 30 seconds the latest check-in summary and `deterioration` data point appear on the screen without manual refresh (driven by the SSE `call_scored` event).
5. The web `/patient` view and the mobile screen, opened side-by-side as the same patient, stay in lockstep — both badges update together, both ringing panels appear together. (This is the demo money shot.)

### 3.4 Open questions to confirm with the backend agent before building §3.1/§3.2

These are items where mobile cannot proceed without a backend decision:

1. **SSE auth for mobile.** Today `/api/stream` uses the cookie session (`web_auth.py`). Mobile holds a JWT, not a cookie. Either (a) accept `Authorization: Bearer` on `/api/stream`, or (b) add `/api/stream/device` that authenticates via `auth.py`'s device JWT and filters events to `patient_id` automatically. Pick one and document in `docs/backend-contract.md`.
2. **Patient self-fetch endpoint.** `GET /api/patients` returns the full list. Mobile only needs its own patient. Either reuse the list endpoint (cheap and fine) or add `GET /api/patients/{pid}` returning a single record. Either is acceptable; just decide.
3. **Rationale URL in pairing.** When pairing displays "this device is being added to the care of *David Patel*," is the patient name returned by `/api/pair/exchange`? It currently returns `patient_id` only. If a friendly name is wanted at pairing time, add it to the response (lock the contract change in `docs/backend-contract.md`).

## Part 4 — What you should NOT do

- **Do not** convert `sentinel-live` to a pnpm/yarn workspaces monorepo. Keep `mobile/` as a sibling of `backend/` and `frontend/`, mirroring the existing flat style.
- **Do not** check in `mobile/android/` or `mobile/ios/`. They're regenerated by `expo prebuild` and patched by the local plugin.
- **Do not** build CallKit/PushKit/foreground-service ringing for the hackathon.
- **Do not** delete `sentinel-live/docs/backend-contract.md`. It's the canonical doc; the mobile copy is a one-line stub pointing back at it.
- **Do not** silently work around SSE auth on the client. If `/api/stream` rejects mobile's JWT, file the backend change, don't pass cookies in a hacky way.
